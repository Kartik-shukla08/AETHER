# Railway Deployment Blueprint

This document details how to deploy the multi-service monorepo onto **Railway**.

## Architecture & Communication Flow

In Railway, services in the same project can communicate privately over the internal network using Railway's internal DNS name mappings (typically in the format `http://<service-name>.railway.internal:<port>`).

```text
               User (Browser)
                     │
         Public HTTPS│
                     ▼
         ┌───────────────────────┐
         │   Frontend Service    │
         │ (Next.js - Port 3000) │
         └───────┬───────┬───────┘
                 │       │
      Public CORS│       │Public WebSockets
     HTTP / SSE  │       │(via Browser)
                 ▼       ▼
         ┌──────────────┐ ┌──────────────┐
         │ Backend API  │ │  Ingestion   │
         │ (Port 4000)  │ │ (Port 4010)  │
         └───────┬──────┘ └───────▲──────┘
                 │                │
                 │                │
          Private│Publish         │Private Ingest
          Redis  │Event           │HTTP POST
          Queue  ▼                │
            ┌─────────┐      ┌────┴────┐
            │  Redis  │ ───> │ Worker  │
            │ Service │      │ Daemon  │
            └────┬────┘      └────┬────┘
                 │                │
                 └──────┬─────────┘
                        │
                        ▼
                  ┌────────────┐
                  │ PostgreSQL │
                  │  Database  │
                  └────────────┘
```

---

## 1. Databases Setup

Create a new Railway project and provision both **PostgreSQL** and **Redis** from the Railway interface:

1. Click **+ New** -> **Database** -> **Add PostgreSQL**.
2. Click **+ New** -> **Database** -> **Add Redis**.

These databases will automatically expose variables like `DATABASE_URL` and `REDIS_URL` to services linked inside the same project.

---

## 2. Deploying Services from the Monorepo

We will deploy four separate services, all linked to the **same** GitHub repository. For each service, you will set the **root directory** to `/` and configure the **Dockerfile Path** in Railway service settings.

### Service A: Backend API

* **Service Name**: `backend`
* **Dockerfile Path**: `apps/backend/Dockerfile`
* **Root Directory**: `/` (must build from repo root to resolve monorepo workspaces)
* **Railway Service Links**: Link with **PostgreSQL** and **Redis** (Railway will automatically inject `DATABASE_URL` and `REDIS_URL`).
* **Environment Variables**:
  * `PORT`: `4000` (automatically exposed publicly by Railway)
  * `HOST`: `0.0.0.0`
  * `NODE_ENV`: `production`

---

### Service B: Ingestion API

* **Service Name**: `ingestion`
* **Dockerfile Path**: `apps/ingestion/Dockerfile`
* **Root Directory**: `/`
* **Railway Service Links**: Link with **PostgreSQL**.
* **Environment Variables**:
  * `PORT`: `4010` (Railway exposes this port; our app binds to `PORT`)
  * `PORT_INGESTION`: `4010`
  * `HOST`: `0.0.0.0`
  * `NODE_ENV`: `production`

---

### Service C: Worker Daemon

* **Service Name**: `worker`
* **Dockerfile Path**: `apps/worker/Dockerfile`
* **Root Directory**: `/`
* **Railway Service Links**: Link with **Redis** (Railway injects `REDIS_URL`).
* **Environment Variables**:
  * `INGESTION_URL`: `http://ingestion.railway.internal:4010/logs` (Uses private internal network)
  * `NODE_ENV`: `production`

---

### Service D: Frontend Web App

* **Service Name**: `frontend`
* **Dockerfile Path**: `apps/frontend/Dockerfile`
* **Root Directory**: `/`
* **Environment Variables**:
  * `PORT`: `3000`
  * `HOST`: `0.0.0.0`
  * `NODE_ENV`: `production`
  * `NEXT_PUBLIC_BACKEND_URL`: `https://backend-production.up.railway.app` (Replace with your backend's **public** domain generated by Railway)
  * `NEXT_PUBLIC_INGESTION_WS_URL`: `wss://ingestion-production.up.railway.app` (Replace with your ingestion service's **public** domain generated by Railway)
  * `NEXT_TELEMETRY_DISABLED`: `1`

> [!NOTE]
> Next.js bundles `NEXT_PUBLIC_` variables at **build time**. Ensure you set the correct public domains for the backend and ingestion services in the Frontend service settings *before* triggering the deployment build of the frontend.

---

## 3. Database Migrations

During the start phase, the `backend` and `ingestion` containers run `npx prisma migrate deploy` automatically. This applies any pending migrations to the PostgreSQL database safely before launching the fastify processes.
